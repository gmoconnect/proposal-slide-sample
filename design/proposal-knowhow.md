## 1. 大前提：官公庁提案書の本質

### 「良い提案」ではなく「高得点の提案」を目指す

官公庁の提案書は、民間営業のプレゼンとは根本的に異なる。評価は**公開された評価基準に基づくスコアリング**で行われ、最高得点の提案が採択される。自社が「良い」と思う提案ではなく、**評価基準上で最も高い点数を取れる提案**を目指すことが最重要原則。

### 3点セットの徹底読み込み

提案書作成の前に、以下の3文書を徹底的に分析する：

| 文書 | 内容 | 注目ポイント |
|------|------|-------------|
| **募集要項** | 参加条件、提出形式、期限 | フォーマット要件、ページ制限、提出方法 |
| **仕様書** | プロジェクトの範囲と技術要件 | 求められる業務内容の詳細 |
| **評価基準書** | 採点方法と配点 | 各項目の配点比重、評価観点の文言 |

加えて、当該自治体の**総合計画**（上位計画）を調査し、プロジェクトが自治体の中長期ビジョンの中でどう位置づけられるかを把握する。

### 評価方式の理解

| 方式 | 計算方法 | 特徴 |
|------|----------|------|
| **加算方式** | 技術点 ＋ 価格点 ＝ 総合点 | 技術力と価格のバランスで勝負 |
| **除算方式** | 技術評価点 ÷ 入札価格 | コストパフォーマンス重視 |

### 典型的な評価構造

| フェーズ | 配点目安 | 評価対象 |
|----------|----------|----------|
| **書類審査** | 〜30点 | 実績、拠点、技術者資格、入札金額 |
| **ヒアリング審査** | 〜100点 | 提案内容、Q&A対応、チーム連携 |

## 2. ストーリー構成

### 2.1 アンサーファースト（結論先行）

**最も重要な原則：結論から述べる。**

Barbara Mintoが McKinsey 在籍時に開発した**ピラミッド原則**がベース。学術論文のように根拠を積み上げて結論に至るのではなく、**結論（推奨アクション）をピラミッドの頂点に置き、その下に根拠を階層的に配置**する。

軍事用語の **BLUF（Bottom Line Up Front）** と同じ考え方。

```
        ┌─────────────┐
        │   結論/提言   │  ← まずここを述べる
        └──────┬──────┘
       ┌───────┼───────┐
       │       │       │
   ┌───┴──┐┌───┴──┐┌───┴──┐
   │根拠1 ││根拠2 ││根拠3 │  ← 支持する理由（MECE）
   └───┬──┘└───┬──┘└───┬──┘
       │       │       │
     詳細    詳細    詳細    ← 具体的データ・事例
```

**各評価項目への回答形式：**
> 「〇〇については、△△を実施します。これにより、□□の効果が見込まれます。」

- 何を実施するか（What）を明示
- それがどう課題解決につながるか（Why/How）を説明
- 断定形「〜します」で記述（「〜したいと考えます」は避ける）

### 2.2 SCQA フレームワーク

提案書全体の物語構造に使える、McKinsey/BCG/Bain共通のフレームワーク：

| 要素 | 内容 | 提案書での例 |
|------|------|-------------|
| **S**ituation（状況） | 評価者が既に知っている前提 | 「〇〇市では人口減少が進行しており…」 |
| **C**omplication（課題） | なぜ今行動が必要か | 「しかし現行システムでは対応が困難に…」 |
| **Q**uestion（問い） | S+Cから自然に生まれる問い | 「いかに効率的に〇〇を実現するか？」 |
| **A**nswer（回答） | あなたの提案＝本提案書の主張 | 「本提案では△△アプローチにより…」 |

> **Tip:** Bain/BCGでは短縮版 **SCR（Situation-Complication-Resolution）** もよく使われる。

### 2.3 顧客理解（Why）→ What → How

提案書の流れは、評価者の思考順序に沿わせる：

```
Why（なぜこの課題が重要か）
  → What（何を提案するか）
    → How（どう実現するか）
```

1. **Why**：自治体が抱える課題の本質を深く理解していることを示す
2. **What**：課題に対する具体的なソリューションを提示
3. **How**：実施体制、スケジュール、品質管理の具体策

### 2.4 評価項目への完全対応

**最もインパクトの大きい戦略。** 評価基準書の文言を精密に分解し、すべての要素に漏れなく回答する。

官公庁の評価文書では、「〜であるとともに」「かつ」などの接続詞で複数の評価ポイントが1文に結合されている。**一見3項目に見える評価基準が、分解すると11の要素を含んでいる**ことがある。

**実践方法：**
1. 評価基準書の各項目を文節単位で分解する
2. 分解した要素ごとに回答を用意する
3. 評価基準書の項目順に提案書の章立てを対応させる
4. 配点が高い項目に比例してページ数・詳細度を厚くする

### 2.5 顧客の用語をそのまま使う

**行政用語は絶対に言い換えない。** 自治体が仕様書で使っている用語をそのまま使用する。「改善」した言い回しにすると、評価者が該当箇所を見つけられず、失点の原因になる。

### 2.6 サマリー → 詳細の階層構造

各セクションは以下の構造で記述する：

```
■ セクションタイトル（評価項目に対応）
  ├── エグゼクティブサマリー（1-2文で結論）
  ├── 概要説明（主要な提案内容を箇条書き）
  └── 詳細説明（具体的手法、データ、事例）
```

### 2.7 スライド間の論理的つながり（アクションタイトル）

各スライドの見出し（アクションタイトル）だけを順番に読んだときに、**提案全体のストーリーが理解できる**ようにする。

| NG（トピックラベル） | OK（アクションタイトル） |
|---------------------|------------------------|
| 「市場環境分析」 | 「〇〇市場は年率5%で縮小しており、早期の対策が不可欠」 |
| 「提案概要」 | 「AIを活用した△△システムにより、処理時間を60%短縮」 |
| 「実施体制」 | 「専任PM配置と週次報告体制で、リスクの早期検知を実現」 |

### 2.8 MECE（漏れなくダブりなく）

Barbara Mintoが1960年代にMcKinseyで考案。提案書のあらゆる分類・分析に適用する：

- **Mutually Exclusive（相互排他）**：カテゴリ間に重複がない
- **Collectively Exhaustive（完全網羅）**：カテゴリで全体をカバーしている

## 3. スライド設計ノウハウ

### 3.1 スライド作成の5ステップ

**いきなりPowerPointを開くのは厳禁。** 以下の順序で段階的に作成する：

| ステップ | 内容 | ツール |
|---------|------|-------|
| **① 言語化** | 伝えたいことをWordで網羅的に書き出す | Word / テキスト |
| **② 構造化** | 箇条書きで主張と根拠を階層化する | Word / テキスト |
| **③ 空パッケージ作成** | スライドタイトルとキーメッセージのみ先に埋める | PowerPoint |
| **④ ボディ制作** | 根拠となるグラフ・チャート・表を作成する | PowerPoint |
| **⑤ 装飾完成** | 脚注・出所・ページ番号を追加し仕上げる | PowerPoint |

> **Tip:** A4横の方眼紙で「紙芝居」として手書きで構想をまとめてからデジタルに移ると、質と生産性を同時に高められる。

### 3.2 スライドの6構成要素

すべてのスライドは以下の6要素で構成される：

```
┌──────────────────────────────────────────────────┐
│ [スライドタイトル] セクション名（複数枚は1/2表記）  │ ← ①タイトル
│ [キーメッセージ] 完全な文でそのスライドの主張を述べる│ ← ②キーメッセージ
├──────────────────────────────────────────────────┤
│                                                  │
│   ┌─────────────────────────────────────────┐    │
│   │                                         │    │
│   │   チャート / テーブル / フレームワーク      │    │ ← ③ボディ
│   │   （キーメッセージを裏付けるエビデンス）     │    │
│   │                                         │    │
│   └─────────────────────────────────────────┘    │
│                                                  │
│  *1) 補足の注釈                                   │ ← ④脚注
│  出所: 〇〇調査（2025年）                 Page XX  │ ← ⑤出所 ⑥ページ番号
└──────────────────────────────────────────────────┘
```

### 3.3 1スライド1メッセージ

1枚のスライドで伝えるメッセージは**厳密に1つ**。2つ以上のメッセージがある場合はスライドを分割する。

### 3.4 キーメッセージ（アクションタイトル）

スライド上部に**完全な文（主語＋述語）** でそのスライドの主張を記述する。単なるトピック名ではなく、**「So What?（だから何？）」への回答**。

#### キーメッセージの検証方法

- **キーメッセージだけを順番に読んで**、提案全体のストーリーが論理的に繋がるか確認する
- ボディ（根拠）とキーメッセージの間に**「Why so?（なぜそう言えるか？）」**と**「So what?（だから何？）」**の関係が成立するか検証する

```
キーメッセージ ←── So what? ──── ボディ（根拠データ）
    │                                    ↑
    └──── Why so? ───────────────────────┘
```

```
┌──────────────────────────────────────────┐
│ AIチャットボット導入により、問合せ対応時間を│  ← アクションタイトル
│ 年間2,000時間削減できる                   │    （完全な文で主張を述べる）
├──────────────────────────────────────────┤
│                                          │
│   ┌─────────┐  ┌─────────┐              │
│   │ 現状    │  │ 導入後  │              │  ← ボディ（根拠データ）
│   │ 5,000h  │→│ 3,000h  │              │
│   └─────────┘  └─────────┘              │
│                                          │
│  出典：〇〇調査（2025年）                  │  ← ソースライン
└──────────────────────────────────────────┘
```

### 3.5 アジェンダスライドの戦略的活用

アジェンダは**単なる目次ではなく、資料内での現在位置を視覚化する役割**を持つ。

- セクション間にアジェンダスライドを挟み、**全体像の中で今どこにいるか**を示す
- スライド上部にプロセス図を配置し、現在のセクションをハイライトする方法も有効
- 特にページ数が多い提案書（20ページ超）では**必須**

```
  [背景]  ──  [課題]  ──  ■提案■  ──  [体制]  ──  [効果]
                           ↑ 現在地
```

### 3.6 情報密度のコントロール

- **メインスライド**：主要メッセージと根拠のみ
- **Appendix**：補足データ、詳細仕様、参考情報
- 1スライドに詰め込みすぎない。迷ったらAppendixへ移動
- **突っ込まれそうなポイント**は本編に含めず、Appendixとして事前準備しておくと意思決定を加速させる

### 3.7 Zの法則（視線誘導）

人の視線は**Z字型**に移動する。最も重要な情報を左上に配置し、結論や行動喚起を右下に置く。

```
① ──────→ ②
         ╱
       ╱
③ ──────→ ④
```

| 位置 | 配置すべき内容 |
|------|---------------|
| ①左上 | スライドタイトル / 最重要メッセージ |
| ②右上 | 補足的な要点 / 数値ハイライト |
| ③左下 | 根拠データ / 図表 |
| ④右下 | 結論 / 行動喚起 |

### 3.8 レイアウトの一貫性

- **グリッドシステム**を使い、全スライドで要素の配置を統一
- マージン、余白、行間を全ページで同一に
- スライドサイズは用途で選択：**A4（印刷用）**、**16:9（投影用）**
- 意図的な余白を恐れない。余白は読みやすさに直結する

### 3.9 ボディの5大レイアウトパターン

ボディ部分の配置は、以下の5パターンに大別される：

```
① 並列配置           ② 順序関係          ③ 分割構造
┌─────┐ ┌─────┐    ┌─────┐            ┌─────┐
│  A  │ │  B  │    │  A  │            │  A  │
└─────┘ └─────┘    └──┬──┘            └──┬──┘
                      ↓              ┌───┴───┐
                   ┌─────┐        ┌──┴──┐ ┌──┴──┐
                   │  B  │        │  B  │ │  C  │
                   └─────┘        └─────┘ └─────┘

④ 複合型（合流）      ⑤ 4分割（2×2）
┌─────┐ ┌─────┐    ┌─────┐ ┌─────┐
│  A  │ │  B  │    │  A  │ │  B  │
└──┬──┘ └──┬──┘    ├─────┤ ├─────┤
   └───┬───┘       │  C  │ │  D  │
    ┌──┴──┐        └─────┘ └─────┘
    │  C  │
    └─────┘
```

| パターン | 用途 |
|---------|------|
| **① 並列配置** | Before/After比較、2案の対比、左にデータ・右に解説 |
| **② 順序関係** | 因果関係、ステップ、上位概念→下位概念 |
| **③ 分割構造** | 1つの要素から複数に展開（ロジックツリー等） |
| **④ 複合型** | 複数の要素が1つの結論に合流する構造 |
| **⑤ 4分割** | 2×2マトリクス、4つの並列要素の整理 |

### 3.10 図表・チャートの体系的活用

#### グラフ5種（定量データの可視化）

| グラフ種類 | 適した内容 | 頻出度 |
|-----------|-----------|-------|
| **棒グラフ / 積み上げ棒グラフ** | 比較、構成比、ランキング | ★★★ 最頻出 |
| **折れ線グラフ** | 時系列推移、トレンド | ★★★ |
| **ウォーターフォール図** | 増減分析、コスト構造の段階的変化 | ★★☆ |
| **散布図** | 相関関係の表現 | ★☆☆ |
| **バブルチャート** | 3次元情報の表現（X軸×Y軸×サイズ） | ★☆☆ |

> **注意：円グラフは基本的に避ける。** 人間は角度の認知が苦手であり、棒グラフの方が正確に比較できる。

#### チャート・図解（構造・関係の可視化）

| チャート種類 | 適した内容 |
|------------|-----------|
| **表（テーブル）** | 詳細な比較、仕様一覧 |
| **2×2マトリクス** | 二軸での分類・ポジショニング（最頻出チャート） |
| **プロセス図（矢印フロー）** | 手順、工程、バリューチェーン |
| **タイムライン / ガントチャート** | スケジュール、ロードマップ、プロジェクト全体像 |
| **カード型レイアウト** | 複数の要点を並列表示（3〜4枚のカード） |
| **Before/After** | 導入効果の可視化 |
| **ロジックツリー** | 課題分解、原因分析 |

#### グラフの補助テクニック

- すべてのチャートに**Callout（注釈吹き出し）**をつけ、データの意味を明示する
- **差分矢印ラベル**を挿入し、ギャップや変化量を数値で示す
- グラフの**タイトル・単位**は必ず表記する
- 軸のスケールや色使いは全ページで一貫性を持たせる
- 国際比較データでは**国旗アイコン**等の視覚的補助が有効

### 3.11 Squint Test / Glance Test

- **Squint Test（BCG）**：目を細めてスライドを見たとき、キーメッセージが視覚的に浮かび上がるか？
- **Glance Test（Duarte）**：3秒でスライドの最重要ポイントが判別できるか？

通過しないスライドはリデザインが必要。

## 4. 説得力の強化

### 4.1 Before/After 比較

導入前後の状態を**同一フォーマット**で対比し、改善効果を一目で示す。

```
┌─────────────────┐     ┌─────────────────┐
│   Before         │     │   After          │
│                  │     │                  │
│ 処理時間: 5日    │ ──→ │ 処理時間: 1日    │
│ エラー率: 15%    │     │ エラー率: 2%     │
│ 年間コスト: 5千万│     │ 年間コスト: 2千万│
└─────────────────┘     └─────────────────┘
                  ▲ 削減効果: 60%
```

### 4.2 定量データの活用

抽象的な表現を避け、**具体的な数値**で語る。

| NG（曖昧） | OK（定量的） |
|------------|-------------|
| 「大幅に改善します」 | 「処理時間を60%短縮します（5日→2日）」 |
| 「多くの実績があります」 | 「過去3年間で同規模案件を12件完遂」 |
| 「コスト削減が見込まれます」 | 「年間3,000万円のコスト削減（ROI: 250%）」 |

### 4.3 客観的根拠の提示

> 「根拠のないパッションは単なる思い込みになってしまう」

- 統計データ、公的機関の調査結果
- 過去の実績データ（類似案件の成果）
- 第三者評価、認証取得実績
- クライアントの声（許可を得た上で）

**論理フロー：** 問題提起 → 現状分析 → 課題抽出 → 提案内容 → 実施計画 → 効果予測

### 4.4 Cialdiniの説得の6原則（提案書への応用）

Robert Cialdiniの心理学研究に基づく、科学的に実証された説得技法：

| 原則 | 提案書での活用方法 |
|------|-------------------|
| **権威性（Authority）** | 有資格者の配置、業界認証、専門機関との連携実績 |
| **社会的証明（Social Proof）** | 同規模自治体での導入事例、類似案件の成功実績 |
| **一貫性（Commitment）** | 自治体の総合計画・公約との整合性を明示 |
| **好意（Liking）** | 地域への理解、共通の課題認識を示す |
| **希少性（Scarcity）** | 自社独自の技術・ノウハウ、他社にない強み |
| **返報性（Reciprocity）** | 提案段階での無償分析・インサイト提供 |

### 4.5 ストーリーテリング

- **実在の成功事例**を物語として紹介し、評価者に「自分たちの自治体でも実現できる」と想像させる
- **Context（背景）→ Characters（登場人物）→ Conflict（課題）→ Resolution（解決）** の4要素
- 抽象的なメリットではなく、**具体的な場面**を描写する

## 5. ビジュアルデザイン（色・フォント・装飾）

### 5.1 配色

#### 基本方針：信頼感・誠実さを伝える配色

```
■ メインカラー        ■ サブカラー         ■ アクセントカラー
  ネイビー/ダークブルー    グレー（濃淡2段階）    1〜2色（強調用）
  #1B3A5C 程度            #666666 / #999999     #E63946（赤系）等
```

| ルール | 説明 |
|--------|------|
| **白背景が基本** | 清潔感・可読性を確保 |
| **メインカラーはネイビー/ダークブルー** | 信頼感・誠実さ・知性を想起させる |
| **使用色は最大3色**（白・黒を除く） | 色が増えると雑多な印象に |
| **アクセントカラーは1〜2色** | 強調したいデータや要点にのみ使用 |
| **色の意味を統一** | 同じ色は全スライドで同じ意味を持たせる |

#### 色の心理効果

| 色 | 印象 | 提案書での用途 |
|----|------|---------------|
| **ネイビー/ダークブルー** | 信頼・誠実・知性 | タイトル、見出し、主要要素 |
| **白** | 清潔・明快 | 背景 |
| **グレー** | 落ち着き・安定 | 補足テキスト、罫線、背景色（薄） |
| **赤/オレンジ** | 注意・重要 | アクセント、警告、重要数値の強調 |
| **グリーン** | 成長・成功 | 正の効果、改善指標 |

### 5.2 フォント

| 項目 | 推奨 |
|------|------|
| **書体** | ゴシック体（メイリオ、游ゴシック、Noto Sans JP） |
| **種類数** | 最大2種類（見出し用＋本文用） |
| **サイズ階層** | タイトル: 24-28pt / 小見出し: 18-20pt / 本文: 14-16pt / 注釈: 10-12pt |
| **最低サイズ** | **16pt以上を推奨。12pt以下は避ける**（可読性が著しく低下） |
| **行間** | 本文の1.4〜1.6倍 |
| **英数字** | 欧文フォント（Arial, Segoe UI等）を併用すると可読性向上 |

> **Tip（McKinsey流）:** 見出しにSerifフォント（Times New Roman, Georgia）を使うと、プレミアム感のある金融レポート調に。本文のラベルや数値はSans-serifで視認性を確保。

### 5.3 太字（Bold）での強調

- 各スライドの**キーメッセージ**を太字にする
- 評価者が**スキャン読み**しても要点が拾える状態を作る
- 太字の使いすぎは逆効果。1スライドで太字にするのは**1〜3箇所**に抑える

### 5.4 Data-Ink Ratio（Edward Tufte）

> 「何よりもまず、データを見せよ。」（Above all else, show the data.）

- **Data-Ink Ratio = データを表すインク量 ÷ 全体のインク量**
- この比率を最大化する。つまり、装飾を削ぎ落としてデータそのものを際立たせる
- 不要なグリッド線、3D効果、背景画像、装飾的な図形を排除（= **Chart Junkの排除**）
- **Small Multiples**：同一フォーマットの小さなグラフを並べて比較を容易にする

### 5.5 その他デザインTips

- **コントラスト**：テキストと背景のコントラスト比を十分に確保（WCAG AA基準: 4.5:1以上）
- **整列（Alignment）**：すべての要素をグリッドに揃える。微妙なズレは不信感を生む
- **近接（Proximity）**：関連する情報をグループ化し、無関係な要素とは距離を置く
- **反復（Repetition）**：デザイン要素（色、形、フォント）を一貫して繰り返す

## 6. 官公庁特有の注意点

### 6.1 民間提案書との決定的な違い

| 観点 | 民間向け | 官公庁向け |
|------|---------|-----------|
| **読み手** | 業界を理解している意思決定者 | 技術に不慣れな管理職を含む非専門家 |
| **冒頭の内容** | 製品の優位性、会社の成長 | 自治体の課題と提案する解決策 |
| **デザイン** | 最小限のテキスト、視覚的インパクト | **図表＋十分な説明文のセット** |
| **用語** | モダンなマーケティング用語 | **公文書の用語をそのまま使用** |
| **図表の扱い** | 図だけで完結 | **必ず説明文を添付**（図だけでは評価されない） |

### 6.2 テキスト重視：図には必ず説明文を添える

官公庁の評価者は、技術の非専門家を含む。**「見ればわかる」は通用しない。** すべてのグラフ・図表・ダイアグラムに、そのポイントを文章で明記する。

### 6.3 自社PRは後回し

- **会社紹介、創業ストーリー、顧客ロゴの羅列は提案書の末尾に**
- 冒頭で自社を語ることは「自己本位」と受け取られ、評価を下げるリスクがある
- 提案書の前半は**自治体の課題認識と解決策**に集中する

### 6.4 Q&A対策：質疑応答が採否を決める

> 多くの企業がプレゼンに全力を注ぎ、Q&Aを補助的と考える。これは致命的な誤り。

**Q&A戦略：**
- **質問の受付担当を1名指名**し、適切な専門家に振り分ける（チーム連携を示す）
- 想定質問リストを事前に作成し、回答担当者を割り当てる
- 全員が最低1回は発言する（チームの実質性を示す）

### 6.5 「提出書類にないことは評価されない」

プレゼンで新しい内容を話しても、**書面に記載がなければ加点されない**。プレゼンは書面の内容を補強・強調する場であり、新規情報を初出しする場ではない。

### 6.6 チーム体制

- **単独でのプレゼンは避ける**（組織的な対応力不足と見なされる）
- 募集要項で指定された上限人数に合わせる
- 各メンバーの役割を明示し、全員が発言する体制を組む

### 6.7 質問期間の戦略的活用

公募の質問期間には積極的に質問を提出する。曖昧な要件を明確化すると同時に、**関与意欲の高さを示すシグナル**にもなる。

## 7. McKinsey/BCG スタイル スライドデザインリファレンス

### 7.1 McKinsey スライド構造

```
┌──────────────────────────────────────────────────┐
│ [Action Title] 完全な文で洞察を述べる              │ ← タイトル
│ [Subtitle] タイトルを補足する説明（任意）           │ ← サブタイトル
├──────────────────────────────────────────────────┤
│                                                  │
│   ┌─────────────────────────────────────────┐    │
│   │                                         │    │
│   │   チャート / テーブル / フレームワーク      │    │ ← ボディ
│   │   （タイトルの主張を裏付けるエビデンス）     │    │
│   │                                         │    │
│   └─────────────────────────────────────────┘    │
│                                                  │
│  Source: 出典情報                        Page XX  │ ← ソースライン
└──────────────────────────────────────────────────┘
```

**McKinseyデザインの特徴：**
- ミニマリスト。派手なグラフィック、アニメーション、3D効果は使わない
- フォントは1〜2種類（Arial + Georgia が標準）
- 色は控えめに使い、重要データの強調にのみ鮮やかな色を使う
- すべてのチャートにCallout（注釈吹き出し）をつけ、データの意味を明示
- 一貫したビジュアルテーマ

### 7.2 BCG の5次元スライドライティング

1. **Purpose（目的）**：各スライドが果たす機能を明確にする（データ伝達 / プロセス説明 / 文脈提供）
2. **Title（タイトル設計）**：ラベルではなく完全な思考を表現するアクションタイトル
3. **Body（本文）**：チャート選択、データ配置、テキストレイアウトの最適化
4. **Emphasis（視覚的強調）**：色、配置、余白を戦略的に使い視線を誘導
5. **Quality（品質管理）**：明確性、正確性、インパクトの検証

**BCGストーリーライニング5ステップ：**
1. ピラミッド原則で基本ロジックを構築
2. 聴衆の視点・知識レベルに合わせて内容を調整
3. エグゼクティブサマリーをバックボーンとして作成
4. アクションタイトルとラフスケッチでスライドに分解
5. 一貫性・完全性・インパクトの体系的レビュー

### 7.3 Duarte S.P.A.C.E. 原則

Nancy Duarteのプレゼンデザイン5原則：

| 原則 | 内容 |
|------|------|
| **S**implify（簡潔化） | メッセージを支えない要素をすべて除去 |
| **P**lan（計画） | PowerPointを開く前に構成を設計 |
| **A**rrange（配置） | 要素を意図的に配置。無秩序な配置は意味を損なう |
| **C**onsistency（一貫性） | 全スライドで視覚的統一感を維持 |
| **E**mphasis（強調） | サイズ・色・配置で最重要ポイントを即座に判別可能にする |

### 7.4 McKinsey風スライド画像生成プロンプト

AI画像生成ツールでMcKinsey風のスライドイメージを作成する際のプロンプト：

## 8. 提案書テンプレート（推奨9セクション構成）

| # | セクション | 内容 | ページ目安 |
|---|-----------|------|-----------|
| 1 | **エグゼクティブサマリー** | 提案全体の要約。結論と主要な提案内容を凝縮 | 1-2ページ |
| 2 | **対象ビジョンとスコープ定義** | 自治体の目指す姿と、本提案がカバーする範囲 | 1-2ページ |
| 3 | **ソリューション提案** | 具体的な解決策の内容と特長 | 3-5ページ |
| 4 | **実施上の重要ポイント** | リスク管理、品質確保、成功のカギとなる要素 | 2-3ページ |
| 5 | **アプローチ・ロードマップ** | 段階的な実施計画とスケジュール | 2-3ページ |
| 6 | **チーム体制と役割分担** | メンバー構成、資格、担当領域 | 1-2ページ |
| 7 | **コスト見積もりと効果予測** | 予算内訳と定量的な効果試算（ROI） | 1-2ページ |
| 8 | **長期戦略計画** | 運用・保守、持続可能性、将来の拡張性 | 1-2ページ |
| 9 | **補足資料（Appendix）** | エビデンス、詳細事例、認証一覧、会社概要 | 必要に応じて |

> **重要：** セクション順序は評価基準書の項目順に合わせて柔軟に入れ替える。評価者のスコアシートと提案書の構成が一致していることが最も重要。

## 9. 提出前チェックリスト

### 9.1 ALLEYOOP 5ポイントチェック

| # | チェック項目 | 確認内容 |
|---|------------|---------|
| 1 | **ストーリー** | 顧客ニーズ → 課題特定 → 解決策 の流れが明確か（6W2H で検証） |
| 2 | **客観的根拠** | すべての主張が統計・数値・実績で裏付けられているか |
| 3 | **顧客共感** | 自治体の課題を深く理解していることが伝わるか（最重要評価ポイント） |
| 4 | **新しい発見** | 自治体が気づいていなかった機会やリスクを提示しているか |
| 5 | **品質** | フォーマット統一、誤字脱字なし、用語の一貫性 |

### 9.2 評価項目突合チェック

- [ ] 評価基準書のすべての項目に対応する記述があるか
- [ ] 配点が高い項目に十分なページ数と詳細度を割いているか
- [ ] 評価基準書の文言（キーワード）が提案書に反映されているか

### 9.3 フォーマット要件チェック

- [ ] 指定のフォント、フォントサイズを使用しているか
- [ ] ページ数制限を遵守しているか
- [ ] 指定の提出形式（紙/電子）と部数を確認したか
- [ ] 提出期限と提出場所を再確認したか

### 9.4 Red Team レビュー

- [ ] 提案書を読んだことがない第三者に模擬評価を依頼したか
- [ ] 評価基準に沿った採点を行い、弱点を特定したか
- [ ] Q&A のシミュレーションを実施したか
- [ ] 指摘事項を反映し、最終版を作成したか

### 9.5 最終確認

- [ ] 提出の1日前に完成させ、当日の不測の事態に備えているか
- [ ] 印刷した状態で全体を通し読みしたか
- [ ] アクションタイトルだけを読んで全体ストーリーが通るか確認したか
